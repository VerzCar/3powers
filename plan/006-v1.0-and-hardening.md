# Plan 006 — Hardening (High-risk self-application) + Brownfield Stage Zero

> **Cold-start note.** If you haven't seen the history: read [`docs/STATUS.md`](../docs/STATUS.md)
> first (what's built, validated against the spec, and why this is next). The spec
> [`3Powers_Spec_v0.2.md`](../specs/3Powers_Spec_v0.2.md) is the source of truth. Plans 001–005 delivered
> v0.1 (trust-spine MVP) and v0.5 (full judiciary); the engine self-applies at the **Standard** tier.

## Context — why this is next

Two things make this the right next step, in this order:

1. **An unmet stated requirement (NFR-006).** The spec requires 3Powers' own trust-spine code to pass
   its own gates at the **High-risk** tier. Today self-application is green only at **Standard** — the
   High-risk **mutation** bar isn't executing (the mutmut 3.x runner doesn't import the src-layout
   package). Closing this makes "3Powers is built with 3Powers at High-risk" actually true, and is the
   cleanest spec-fidelity win available.
2. **The highest-value v1.0 feature (Brownfield Stage Zero, §12).** Most teams don't start clean.
   Report-only + ratchet-on-changed-code + characterization tests is what lets 3Powers spread through an
   existing repo without a stop-the-world migration — the gateway to real adoption.

Direction is otherwise on-track (judiciary-first, §17 phasing followed, self-applied). The deeper
approximations — **structural** oracle independence and Spec Kit headless-dispatch/catalog packaging
(A1/A3) — are bigger and are sequenced **after** this plan (see STATUS §6); they are called out here so
the next author doesn't lose them.

## Scope

**In:**
- **A — Close NFR-006:** make the **mutation** gate execute on the trust-spine modules and meet the
  High-risk threshold; add per-path tier scoping so the trust spine is gated at High-risk without forcing
  95% coverage on orchestration code.
- **B — Brownfield Stage Zero (FR-051/052):** a **report-only** mode and **diff-scoped** gating so only
  new/changed code is held to the full process; **characterization-test** reconstruction (FR-053) as a
  command + lighter support.

**Out (→ plan 007+):** observe/feedback loop (§13, FR-054/055), emergency/deviation paths (§14,
FR-056/057), **structural oracle independence** via Spec Kit headless dispatch (FR-021/062), **catalog
distribution** as a Spec Kit extension/preset (A1), a **third adapter**, defect-flow (FR-008) & design
oracles (FR-009), the model-driven eval layer (FR-050), cross-platform validation (NFR-003), root
`LICENSE` (NFR-012).

## Decisions (proposed — revisit if you find better)

| Area | Proposal | Rationale |
|---|---|---|
| **Mutation runner** | Fix mutmut 3.x for the src-layout (ensure the mutated `threepowers` is importable, not shadowed by the editable install — e.g. run in an isolated env / set the mutants path); if it stays brittle, switch to **cosmic-ray** | The spec names the mechanic (mutation), not the tool; pick whichever reliably runs on the src layout. |
| **Per-capability tiers** | Add `--paths`/scoping to `gate run` so diff-coverage + mutation can target the trust-spine files (`canonical`, `keys`, `ledger`, `verify`) at High-risk, while orchestration stays Standard | §4 assigns tiers **per capability area**, not per whole repo; this is the faithful way to satisfy "trust-spine at High-risk." |
| **Report-only / ratchet** | A `--report-only` flag (and/or a tier/config switch) that emits the verdict but does not fail the run or block `advance`; ratchet = flip to blocking on changed code | FR-052 — lets legacy debt not wall off every merge during adoption. |
| **Brownfield diff-scope** | Reuse the existing `--base` + `covdiff.changed_files` to restrict gates to changed files | FR-051 — hold only new/changed code; existing code untouched until modified. |

## Workstreams

1. **Mutation on the trust spine (NFR-006, FR-031).** Get `uv run mutmut run` (scoped to the four
   trust-spine modules — config already in `engine/pyproject.toml`) to actually execute; close surviving
   mutants by adding assertions until `mutation_score ≥ 70`. Wire the result into the `mutation` gate so
   `3pwr gate run … --tier High-risk --mutation` reflects it.
2. **Per-path High-risk scoping.** Add a `--paths`/`--only` option (or read `capabilities` from
   `risk-tiers.yaml` + a file→capability map) so a High-risk run targets the trust-spine files. Verify:
   `3pwr gate run --path engine --tier High-risk --mutation --paths src/threepowers/{canonical,keys,ledger,verify}.py`
   is **green** (≥95% diff-coverage + mutation on those modules), while a Standard run still covers the
   whole engine.
3. **Report-only mode (FR-052).** `3pwr gate run --report-only` emits the normalized verdict and ledger
   entry but returns success and does not block `advance`; document the ratchet to blocking.
4. **Diff-scoped gating (FR-051).** Ensure the suite can be limited to changed files (extend the gates
   that currently run whole-project — tests/conformance — to honor a changed-file scope when given).
5. **Characterization tests (FR-053).** `/3pwr.characterize` (Copilot command) + `3pwr characterize`
   that scaffolds characterization tests pinning a legacy module's current behavior as its oracle;
   reconstruct a minimal spec stub under `specs/`.
6. **Self-application + tests + docs.** Keep the engine green; the headline check is the **High-risk
   self-application** run going green. Update `docs/STATUS.md` (flip NFR-006 to ✅), `CLAUDE.md`,
   `AGENTS.md`, and add new tests citing FR-051/052/053. Commit on `plan-006-*`.

## How to work here (conventions, for a fresh start)

- **The spec is the law.** Validate every change against `3Powers_Spec_v0.2.md`; respect §17 phasing.
- **Keep the engine green under its own gates** (`ruff`/`mypy`/`pytest`, and `3pwr gate run --path engine`).
  Trust-spine modules (`canonical`, `keys`, `ledger`, `verify`) are High-risk — hold coverage ≥95%.
- **No inline gate suppressions** in the engine — the `gate_gaming` gate will (correctly) flag a
  `# type: ignore` / `# noqa` / `pragma: no cover`; fix the underlying issue instead.
- Each new test should reference the FR id it exercises (the spec-conformance gate traces them).
- Add new requirements you implement to the engine's scoped spec `specs/002-engine-trust-spine/spec.md`.
- One plan per slice in `plan/NNN-*.md`; commit on a `plan-NNN-*` branch.
